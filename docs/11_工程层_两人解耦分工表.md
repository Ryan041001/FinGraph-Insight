# 10 两人解耦分工表

## 1. 分工原则

两个人的工作要尽量解耦，核心做法是“API 契约先行”：

- 后端先提供 OpenAPI 文档和 mock JSON。
- 前端先基于 mock JSON 开发页面。
- 后端实现真实逻辑后，只要接口结构不变，前端不需要大改。
- 两人尽量不同时修改同一类文件。

## 2. 角色划分

### 人员 A：后端 / 算法 / 数据

主要负责：

- 数据清洗。
- Neo4j schema 和入库。
- LLM 抽取。
- 实体消歧。
- GraphRAG。
- Text2Cypher 安全校验。
- 定时更新。
- 质量评测。
- 图谱驱动智能研判后端接口。
- K 线行情后端接口。

### 人员 B：前端 / 展示 / 汇报

主要负责：

- Vue3 项目。
- 图谱浏览。
- 企业画像。
- 实时抽取页面。
- GraphRAG 和 Text2Cypher 页面。
- 定时更新状态页。
- Dashboard。
- 智能研判页面。
- K 线详情抽屉和行情图组件。
- PPT、演示脚本和报告系统部分。

## 4. 关键接口交接物

A 需要尽早给 B：

- OpenAPI 地址：`http://localhost:8000/docs`。
- 每个页面需要的 mock response。
- 错误响应格式。
- 图谱节点和边的统一结构。
- 演示用公司名、新闻文本、问答样例。

B 需要尽早给 A：

- 页面需要哪些字段。
- 图谱组件对节点/边格式的要求。
- 哪些接口需要 loading 和进度状态。
- 哪些错误需要前端特殊展示。

## 7. 每日同步机制

每天同步 15 分钟即可，重点只看阻塞项：

1. 今天接口结构有没有变化。
2. 前端是否缺字段。
3. 后端是否缺演示样例。
4. 是否有功能要降级。
5. 是否影响 10 分钟演示路径。

不建议每天讨论大而全的功能想法，避免范围失控。

## 8. 联调检查表

### API 契约检查

- [ ] 每个接口都有请求示例。
- [ ] 每个接口都有成功返回示例。
- [ ] 每个接口都有失败返回示例。
- [ ] 图谱节点和边结构统一。
- [ ] 错误码统一。

### 前端联调检查

- [ ] 后端未启动时有错误提示。
- [ ] 接口 loading 状态明确。
- [ ] 空结果有空状态。
- [ ] 图谱数据过多时有限制。
- [ ] Text2Cypher 安全拒绝能展示原因。

### 后端联调检查

- [ ] CORS 已配置。
- [ ] 接口返回字段稳定。
- [ ] 低置信度数据不会默认入库。
- [ ] Text2Cypher 只读限制有效。
- [ ] 定时任务失败不会影响主服务。

## 9. 最容易冲突的地方

| 冲突点 | 预防方式 |
| --- | --- |
| 前端想要字段后端没有 | Day 2 固定 mock response，字段变更先同步 |
| 后端改接口导致前端坏 | API 版本化或保持兼容字段 |
| 图谱节点边格式不统一 | 统一使用 `GraphNode` 和 `GraphEdge` |
| 演示样例临时找不到 | Week 2 前确定演示数据 |
| 时间不够 | 只保 P0，P1 按完成度加入 |

## 10. 最终责任边界

A 对以下结果负责：

- 基础图谱能入库。
- 抽取接口能返回合法结构。
- 实体消歧和置信度过滤可用。
- GraphRAG 和 Text2Cypher 后端可用。
- 定时更新可手动触发和自动运行。
- 评测指标可计算。
- 智能研判接口如进入 P1，则每条结论有证据来源。
- K 线接口如进入 P1，则支持 A 股日 K 和本地缓存兜底。

B 对以下结果负责：

- 前端页面完整。
- 图谱展示清晰。
- 抽取、问答、更新状态流程可演示。
- PPT 逻辑清楚。
- 报告系统展示部分完整。
- 演示脚本稳定。
- 智能研判页面如进入 P1，则明确展示免责声明。
- K 线详情如进入 P1，则只作为 Company 节点详情，不作为独立主页面。

共同负责：

- 10 分钟演示闭环。
- Docker Compose 能在演示机器启动。
- 报告和 PPT 内容一致。
